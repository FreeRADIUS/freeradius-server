= CHAP

The Challenge Handshake Authentication Protocol (CHAP) authenticates PPP clients without sending the user's password over the wire.  The xref:reference:raddb/mods-available/chap.adoc[mods-available/chap] configuration file specifies the parameters used by the CHAP module.  This document explains how to configure and test CHAP authentication in FreeRADIUS.

== CHAP Authentication Flow

CHAP authentication happens in three stages. The NAS sends a challenge to the client. The client responds with a hashed password and challenge. FreeRADIUS checks the hash and returns an accept or reject message to the NAS, which then grants or denies the session. More details are shown in the net section.

*Phase 1*

The NAS sends a random challenge to the supplicant.  The client performs some computations and returns the result.  The plaintext password never travels on the wire. The NAS packages the result into an `Access-Request` (UDP/1812) containing`User-Name`, `CHAP-Challenge`, and `CHAP-Password`, and forwards it to the FreeRADIUS server.

*Phase 2*

FreeRADIUS processes the request in two steps.  During `recv Access-Request`, the `chap` module detects `CHAP-Password` and sets `Auth-Type := CHAP`.

A database module (such as `files`, `sql`, or `ldap`) looks up the user and adds `Password.Cleartext` into the control list. During `authenticate chap` process, the `chap` module recomputes the hash `MD5(id 竏･ cleartext 竏･ challenge)` locally and byte-compares the result against `CHAP-Password`. The module returns `ok` for a match and `reject` for a mismatch.

*Phase 3*

The NAS receives a single RADIUS reply: `Access-Accept` on success, or `Access-Reject` on failure.  The NAS then accepts or rejects the supplicant's session accordingly.

== Edit mods-available/chap

The xref:reference:raddb/mods-available/chap.adoc[mods-available/chap] module
contains no configuration items, and does not need to be edited.

== Enable mods-available/chap

The `chap` module is enabled by creating a soft link from the
`mods-enabled/` directory to the `mods-available/` directory.

[source,shell]
----
cd raddb/mods-enabled && ln -s ../mods-available/chap
----

The default installation will automatically enable the `chap` module.
In most circumstances, no additional work is required.

== Configure the Virtual Server

The `chap` module must be referenced in two sections of the virtual
server.  Both sections are present in the default
`sites-available/default` virtual server.

=== Configure `recv Access-Request`

The `chap` module must be called in the `recv Access-Request { ... }`
section.  When a request arrives containing a `CHAP-Password`
attribute, the module sets `Auth-Type := ::CHAP`, which causes the
server to call the `authenticate chap { ... }` section later in
processing.

[source,unlang]
----
recv Access-Request {
    ...
    chap
    ...
}
----

=== Configure `authenticate chap`

The `chap` module must also appear in the `authenticate chap { ... }`
section.  This section is called when `Auth-Type` is set to `CHAP`.
The module retrieves the `Password.Cleartext` that has been placed
into the request by a database module (such as `files`, `sql`, or
`ldap`), and verifies the CHAP credentials.

[source,unlang]
----
authenticate chap {
    chap
}
----

The `chap` module requires access to a cleartext password.
`Password.Cleartext` must be set by a prior module in the
`recv Access-Request { ... }` section before the `authenticate chap {
... }` section is reached.  Standard Unix system authentication or passwords  are not compatible with CHAP.

See the xref:concepts:protocol/authproto.adoc#Proto-Password-Compat[protocol compatibility chart] for more information.

== Disable CHAP

The local system requirements may be that CHAP authentication is
forbidden.  In that case, the `chap` module should be removed from the
`recv Access-Request { ... }` section, and the `authenticate chap {
... }` section should be deleted.  The server will then be unable to
process CHAP authentication requests.  All `Access-Request` packets
that contain a `CHAP-Password` attribute will then result in an
`Access-Reject`.

== Testing

Confirm that your server configuration is accurate before proceeding to test CHAP functionality. For more details and guidance, please refer to xref:modules/chap/testchap.adoc[testing].
